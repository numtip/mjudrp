# SharePoint List Strategy

**Status:** BLUEPRINT — 2026-07-06

---

## Background

SharePoint Lists can act as operational metadata tables — lightweight reference data that staff can view and edit without touching JSON files or git. However, they must NOT become a replacement for git-based registry data.

## Evaluation

### Recommended for SharePoint Lists

| List | Purpose | Recommended | Rationale |
|------|---------|-------------|-----------|
| **DRP Categories** | Operational category reference for library metadata columns | ✅ Yes | Staff need to pick from available categories when uploading |
| **DRP Projects** | Active consumer project reference | ✅ Yes | Staff need to know which projects a document belongs to |
| **DRP Owners** | Department and owner reference | ✅ Yes | Staff need to select document owners |
| **Metadata QA Queue** | Documents pending metadata review | ✅ Yes | Track documents needing AI-assisted or human metadata review |

### NOT Recommended for SharePoint Lists

| List | Reason Not Recommended |
|------|----------------------|
| **Evidence Map** | Evidence is complex cross-referencing data better managed in git-controlled JSON. SharePoint Lists lack the validation and cross-reference capabilities of AJV |
| **Relationship Map** | Relationships are auto-generated by the population script. Manual management via SharePoint would introduce inconsistencies |
| **Registry Review Queue** | Registry release decisions should be git-based (pull requests). SharePoint adds a parallel process |

## MVP Approach

### Phase 1 — Manual (Sprint 3B)

Create 3 SharePoint Lists manually via the SharePoint UI:

1. **DRP Categories** — sourced from `registry/categories.sample.json`
2. **DRP Projects** — sourced from `registry/projects.sample.json`
3. **DRP Owners** — sourced from `registry/owners.sample.json`

These lists act as **reference guides** for staff filling metadata columns during upload. They are NOT authoritative — git is authoritative.

### Phase 2 — Read-Only Sync (Future Sprint with Graph)

Once Microsoft Graph read access is configured:

1. Export categories from SharePoint List to validate against registry
2. Optionally import SharePoint List data as registry updates via PR
3. Never auto-write from SharePoint to registry without human review

### Anti-Patterns to Avoid

| Anti-Pattern | Why |
|-------------|-----|
| Using SharePoint Lists as the primary registry | Git is the source of truth. SharePoint Lists are operational references only |
| Building a custom backend to sync Lists ↔ registry | Adds complexity. Use Graph read-only sync or manual export |
| Making Lists required for document upload | Lists can have sync delays. Document upload should not depend on list availability |
| Storing evidence or relationships in Lists | These require cross-reference validation that SharePoint cannot provide |

## Metadata QA Queue List Design

| Column | Type | Required | Description |
|--------|------|----------|-------------|
| Document ID | Single line text | ✅ | DRP Document ID |
| Title | Single line text | ✅ | Document title |
| Status | Choice | ✅ | Pending, In Review, Approved, Rejected |
| Reviewer | Person or Group | ❌ | Assigned reviewer |
| Notes | Multiple lines text | ❌ | Review comments |
| AI Suggestions | Multiple lines text | ❌ | Auto-generated metadata suggestions |
| Created | Date and Time | ✅ | Auto |
